# 电商智能体工作区 V1 执行指南

> 适用对象：Claude Code、GLM-5 等 AI 编码工具
>
> 项目：`growpilot-v2`
>
> 最后更新：2026-04-10

---

## 一、文档目的

本文档不是产品讨论稿，而是面向 AI 编码工具的执行指南。

目标是指导执行“电商智能体右侧区域 V1 重构”这项迭代，确保实施过程中：

- 目标清晰
- 边界明确
- 前后端分工可落地
- 文件级改动范围可控
- 不引入不必要的新技术栈
- 可以在当前仓库基础上稳妥推进

---

## 二、迭代背景

当前电商智能体页面位于：

- `app/dashboard/ecom-agent/EcomAgentPageClient.tsx`

右侧区域当前由：

- `app/dashboard/ecom-agent/MediaGallery.tsx`

负责展示。其现状更接近“素材展示区”，而不是“AI 生成内容的可视化工作区”。

当前主要问题：

1. 右侧区域只平铺图片和视频，缺少工作上下文。
2. 用户无法明确知道当前这轮创作的主目标、主资产和任务状态。
3. 图片、视频、上传参考图、失败任务之间没有清晰结构。
4. 用户无法在右侧自然地继续下一步创作，只能回到左侧聊天区自行组织。
5. 右侧区域没有承接“对话驱动创作工作流”的能力。

因此，V1 的目标不是优化样式，而是把右侧区域升级为可用的创作工作区。

---

## 三、V1 目标定义

V1 的目标是：在不引入新技术栈、不重构后端架构的前提下，将右侧区域从“素材展示区”升级为“可工作的创作工作区”。

V1 必须解决以下四个问题：

1. 用户能看懂当前这轮创作的上下文。
2. 用户能明确看到当前主资产。
3. 用户能区分生成中、已完成、失败的任务状态。
4. 用户能从右侧直接发起下一步动作，而不必回聊天记录中自行整理。

---

## 四、V1 范围

### 4.1 本期纳入

- 工作区头部 `Workspace Header`
- 主资产展示区 `Primary Canvas`
- 任务时间线 `Job Timeline`
- 候选素材区 `Asset Rail`
- 三个主动作：
  - 继续生成
  - 发送回对话
  - 加入项目

### 4.2 本期不纳入

- 后端持久化重构
- 版本树或复杂版本链
- 多人协作
- 跨设备同步
- 新状态管理库
- 大规模后端接口改造
- 全量项目系统重构

## 五、技术策略

### 5.1 技术栈要求

本期不新增技术栈，继续使用现有技术方案：

- Next.js 16.2
- React 19
- TypeScript
- Tailwind CSS 4
- 现有 hooks 与本地状态管理方式

### 5.2 状态管理策略

V1 不引入 Zustand、Redux 或其他新状态库。

优先方案：

- 在 `EcomAgentPageClient.tsx` 中建立一层“工作区视图状态”
- 通过局部组件拆分降低复杂度
- 必要时抽出本地 hook，但不引入新依赖

### 5.3 后端策略

V1 以后端最小配合为原则：

- 不新增核心业务接口作为前提条件
- 优先复用当前 `lib/ai.ts` 路径
- 后端重点保障现有生成链路稳定性

如果当前图片或视频生成结果链路不稳定，应先修复稳定性问题，再进入联调阶段。

## 六、信息架构

右侧区域应调整为如下结构：

### 6.1 Workspace Header

显示当前线程的工作上下文，建议包含：

- 当前线程标题
- 当前模式
- 当前平台
- 当前任务类型
- 当前整体状态
- 当前线程产出数量

### 6.2 Primary Canvas

右侧主舞台，用于展示当前主资产。其职责是：

- 只聚焦一个当前主资产
- 支持图片或视频
- 显示主状态与主动作

### 6.3 Job Timeline

展示当前线程的任务流，至少支持：

- 已提交
- 生成中
- 已完成
- 失败

### 6.4 Asset Rail

展示当前线程下的候选结果与参考素材，包括：

- 图片结果
- 视频结果
- 上传参考图
- 其他可回看结果

## 七、推荐的前端对象模型

为避免继续将 UI 直接绑定在当前松散的 `mediaRefs + currentMedia` 结构上，V1 建议在前端增加一层“视图映射模型”。

建议使用以下概念：

### 7.1 CurrentBrief

表示当前这轮创作任务的上下文，可由当前线程与筛选状态推导得到。

建议字段：

- `threadId`
- `threadTitle`
- `mode`
- `platform`
- `intent`
- `language`
- `status`

### 7.2 WorkspaceJob

表示一次生成任务。

建议字段：

- `id`
- `type`
- `status`
- `label`
- `sourceMessageId`
- `linkedAssetId`
- `createdAt`
- `errorMessage`

### 7.3 WorkspaceAsset

表示右侧工作区中的一个可视资产。

建议字段：

- `id`
- `type`
- `status`
- `dataUrl`
- `prompt`
- `sourceMessageId`
- `createdAt`
- `isUploaded`

### 7.4 SelectedAssetId

表示当前主资产。

规则建议：

- 最新成功生成的资产默认设为主资产
- 用户点击任意素材可切换主资产
- 若主资产被删除，自动回退到最近一个可用资产

---

## 八、建议的组件拆分

当前右侧的 `MediaGallery.tsx` 不宜继续承担整个工作区职责。

建议拆分为以下结构：

- `EcomWorkspacePanel`
- `WorkspaceHeader`
- `WorkspacePrimaryCanvas`
- `WorkspaceJobTimeline`
- `WorkspaceAssetRail`
- `WorkspaceEmptyState`

可选的下一层拆分：

- `WorkspaceAssetCard`
- `WorkspaceJobItem`
- `WorkspacePrimaryActions`
- `WorkspaceAssetMeta`

### 8.1 组件职责建议

#### `EcomWorkspacePanel`

右侧总容器，负责：

- 接收当前线程的资产、任务、选中态
- 管理主区域布局
- 协调子组件

#### `WorkspaceHeader`

负责展示当前线程的上下文信息，不承载生成逻辑。

#### `WorkspacePrimaryCanvas`

负责展示当前主资产及其操作区。

#### `WorkspaceJobTimeline`

负责展示任务状态，不负责素材预览。

#### `WorkspaceAssetRail`

负责展示候选资产列表，不承担主画布职责。

## 九、用户主流程

V1 需要重点保证以下流程成立：

### 9.1 文案 -> 出图 -> 继续生成

1. 用户在左侧发起需求
2. 生成图片
3. 最新图片自动进入主画布
4. 用户从右侧点击“继续生成”
5. 输入区自动获得对应上下文

### 9.2 文案 -> 出视频 -> 查看状态 -> 完成播放

1. 用户发起视频相关需求
2. 任务出现在时间线
3. 生成中状态可见
4. 完成后视频自动成为主资产或进入候选区

### 9.3 上传参考图 -> 图像编辑 -> 结果进入主画布

1. 用户上传图片
2. 右侧出现上传参考素材
3. 后续编辑结果进入工作区

### 9.4 左侧消息 -> 右侧定位

1. 用户点击左侧带资产引用的消息
2. 右侧跳转到关联主资产

## 十、三项核心动作定义

V1 只做三项最重要的动作。

### 10.1 继续生成

作用：将当前主资产作为下一轮创作的参考上下文。

V1 行为建议：

- 将当前资产信息注入输入区上下文
- 根据资产类型选择是否切换模式
- 优先实现“生成同类结果”而非复杂链式编辑

### 10.2 发送回对话

作用：让用户以当前主资产为上下文继续追问。

V1 行为建议：

- 不是直接上传文件到后端
- 而是先将资产引用写回输入区上下文提示
- 让用户继续基于该资产提出修改要求

### 10.3 加入项目

作用：将当前结果沉淀到现有项目体系。

V1 行为建议：

- 优先复用现有本地项目存储逻辑
- 不以新建后端项目接口为前提

## 十一、前后端工作量占比

V1 预计工作量占比：

- 前端：约 75%
- 后端：约 25%

### 11.1 前端职责

- 重构右侧区域的信息架构
- 建立工作区视图状态模型
- 建立主资产机制
- 抽离任务时间线
- 接入三项核心动作
- 完成空态、失败态、加载态设计
- 建立左侧对话与右侧工作区的联动

### 11.2 后端职责

- 确保现有 GenAI 链路稳定
- 明确错误返回与可识别状态
- 不要求在 V1 阶段重做后端数据模型

## 十三、推荐执行顺序

按以下顺序执行，风险最低。

### 阶段 1：冻结边界

先确认：

- V1 模块范围
- V1 三个核心动作
- V1 不做事项

### 阶段 2：建立视图状态模型

先整理前端内部对象模型，再改 UI。

### 阶段 3：拆分右侧结构

完成：

- Header
- Primary Canvas
- Asset Rail

此阶段可先不接完整任务流。

### 阶段 4：接入 Job Timeline

把视频生成中与失败状态抽出。

### 阶段 5：接入三项动作

完成：

- 继续生成
- 发送回对话
- 加入项目

### 阶段 6：联调与真实验证

必须通过真实页面操作验证，不应只依赖静态代码阅读。

## 十四、推荐给 AI 编码工具的分工方式

本项目属于典型的“vibe coding + 结构化收口”场景。
建议不要让单个 AI 工具同时重写整个页面，而应拆成明确的、可验证的子任务。

### 14.1 推荐分工

#### Claude Code

适合负责：

- 结构拆分
- 状态模型整理
- 主文件重构
- 关键交互闭环

#### GLM-5

适合负责：

- 子组件 UI 落地
- 空态/加载态/失败态
- 样式层细化
- 文案与轻交互补全

### 14.2 推荐执行方式

不要让多个工具同时修改同一文件。

推荐拆分方式：

- Claude Code 负责：
  - `EcomAgentPageClient.tsx`
  - `EcomWorkspacePanel.tsx`
  - 状态模型与事件联动

- GLM-5 负责：
  - `WorkspaceHeader.tsx`
  - `WorkspacePrimaryCanvas.tsx`
  - `WorkspaceJobTimeline.tsx`
  - `WorkspaceAssetRail.tsx`
  - 空态与视觉细节

如果只能使用一个 AI 工具，则严格按“先状态、后结构、再样式、最后联动”的顺序执行。

---

## 十五、禁止事项

执行 V1 时，禁止出现以下行为：

- 直接引入新的状态管理库
- 直接重写整个页面且不做结构拆分
- 未理解现有 `mediaRefs` 与线程关系就重做数据结构
- 未经过真实页面验证即声称流程可用
- 将 V2 持久化能力混入 V1
- 在未确认后端稳定性的前提下修改大量接口调用逻辑

---

## 十六、验收标准

V1 完成后，必须满足以下标准：

1. 右侧区域不再只是平铺素材，而具备明确工作区结构。
2. 新生成结果会自动进入当前主画布。
3. 用户可以主动切换当前主资产。
4. 生成中的任务、失败任务可被明确看见。
5. 用户可从右侧直接发起“继续生成”。
6. 左侧消息可以联动右侧对应资产。
7. 至少以下四条流程能够稳定走通：
   - 文案 -> 出图 -> 继续生成
   - 文案 -> 出视频 -> 查看状态 -> 播放
   - 上传参考图 -> 图像编辑 -> 结果进入主画布
   - 左侧消息 -> 右侧定位

---

## 十七、建议的实施输出

执行过程中，建议 AI 工具至少输出以下产物：

1. 代码改动
2. 受影响文件列表
3. 关键状态流说明
4. 已验证流程清单
5. 未解决问题与残余风险

如果当前轮只完成结构重构而未完成联调，应明确说明未完成项，不应将未验证内容描述为已完成。

---

## 十八、结论

V1 的本质不是“把右侧区域做得更丰富”，而是建立一个真正可工作的“对话驱动创作工作区”。

执行时必须坚持以下原则：

- 先明确结构，再调整状态，再接入动作，最后做真实验证
- 优先前端落地，后端仅做最小配合
- 不做超出 V1 范围的系统性扩张
- 所有判断以当前代码与真实运行结果为准

本次迭代完成后，电商智能体页面应从“聊天 + 素材展示”提升为“聊天驱动的创作工作台”。
