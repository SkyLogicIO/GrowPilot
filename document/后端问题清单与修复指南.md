# GrowPilot 后端问题清单与修复指南

> **基于仓库**：`growpliot-server`（manage-server Go + ai-server Python）
> **审查日期**：2026-04-08
> **审查范围**：全量源码逐行比对（Go 路由、Go 验证器、Go-Python 通信格式、Python 端点实现）

---

## 重要说明：ai-tools 是旧路径，GenAI 代理是主路径

后端有两条 AI 对接路径：

- **GenAI 代理**（`/api/v1/genai/v1beta/*`）：完全对标 Google Gemini API，Go 网关透明转发请求，前端 `@google/genai` SDK 只需配置 Base URL + JWT header 即可使用。**这是推荐的主路径。**
- **ai-tools**（`/api/v1/ai-tools/*`）：旧的异步任务系统，有自定义的请求/响应格式、Go 验证器、Python 实现和任务轮询机制。

**本文档中的问题严重度已根据路径重新评估**：仅影响 ai-tools 旧路径的问题已降级为 🟢，因为 GenAI 代理已覆盖相同功能，不受这些问题影响。

以下问题在 ai-tools 路径下是严重阻塞，但**GenAI 代理路径完全不受影响**。如果不再使用 ai-tools 旧路径，这些问题可以不修复。

### #1 image-to-image 路由未在 Go 网关注册 🟢

**原严重度**：🔴 → **现严重度**：🟢（GenAI 代理通过 `generateContent` 覆盖图生图）

**现象**：前端调用 `POST /api/v1/ai-tools/image-to-image` 返回 404。

**原因**：Go 网关注册了 5 个 AI 工具路由，缺少 `image-to-image`。

**位置**：`manage-server/cmd/server/main.go` L166-185

```go
// 当前注册（L170-179）
aiTools.POST("/text-to-image", aiToolsHandler.CreateTextToImageTask)
aiTools.POST("/image-to-video", aiToolsHandler.CreateImageToVideoTask)
aiTools.POST("/text-to-video", aiToolsHandler.CreateTextToVideoTask)
aiTools.POST("/inpaint", aiToolsHandler.CreateInpaintTask)
aiTools.POST("/frame-to-video", aiToolsHandler.CreateFrameToVideoTask)
// ❌ 缺少 image-to-image
```

Python 端已实现（`ai-server/app/api/v1/ai_tools.py` L116）。

**影响范围**：仅 ai-tools 旧路径。GenAI 代理通过 `POST /genai/v1beta/models/gemini-2.5-flash-image:generateContent` 完全覆盖图生图功能。

**修复**（如仍需 ai-tools 路径）：

1. `main.go` L179 后新增：
```go
aiTools.POST("/image-to-image", aiToolsHandler.CreateImageToImageTask)
```

2. `ai_tools_handler.go` 新增 `CreateImageToImageTask` handler（参考 `CreateTextToImageTask` 的结构）。

3. `python_ai_client.go` 新增 `CreateImageToImage` 方法，请求 `/api/v1/ai-tools/image-to-image`。

### #2 Go-Python AI 工具响应格式不匹配 🟢

**原严重度**：🔴 → **现严重度**：🟢（仅影响 ai-tools 路由，GenAI 代理直接透传 Google API 格式）

**现象**：所有 AI 工具（text-to-image、text-to-video、image-to-video、inpaint、frame-to-video）调用成功后，前端拿到的 `data` 中没有任何有用字段（`task_id`、`result_url` 等全部丢失）。

**原因**：Go 期望 Python 返回三层包装格式，但 Python 实际返回扁平 dict。

**Go 期望**（`python_ai_client.go` L122-126）：
```go
type AIToolResponse struct {
    StatusCode int                    `json:"status_code"`
    Message    string                 `json:"message"`
    Data       map[string]interface{} `json:"data"`
}
```

**Python 实际返回**（`ai_tools.py` L67-93）：
```python
return {
    "task_id": "task_xxx",
    "status": "completed",
    "model_name": "nanobanna",
    "result_url": "https://...",
    "created_at": "2026-04-01T12:00:00Z",
    "latency_ms": 1234,
}
```

**解码结果**：Go 解析时 `status_code`=0、`message`=""、`data`=nil，Python 返回的字段全部被丢弃。Go 再把这个空壳包一层返回给前端：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "status_code": 0,
    "message": "",
    "data": null
  }
}
```

**影响链路**（仅 ai-tools 路径）：
```
前端 → Go 网关 → Python AI 层（正确生成结果）→ 返回扁平 dict
       → Go 解码为空 AIToolResponse → 前端拿到空 data
```

**影响范围**：仅 ai-tools 路径。GenAI 代理不经过 Go-Python 通信链路，直接透传 Google API 的响应格式。

**修复方案**（二选一，如仍需 ai-tools 路径）：

**方案 A（推荐）— 修改 Python 端统一包装**：

修改 `ai_tools.py` 所有端点的返回值，统一包裹为：
```python
return {
    "status_code": 0,
    "message": "success",
    "data": {
        "task_id": task_id,
        "status": "completed",
        "model_name": request.model_name,
        "result_url": result.get("result_url", ""),
        "created_at": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
        "latency_ms": result.get("latency_ms", 0),
    }
}
```

需改动的 Python 端点：`text_to_image`（L67）、`image_to_image`、`image_to_video`、`text_to_video`、`inpainting`、`frame_to_video`。

**方案 B — 修改 Go 端直接解码扁平 dict**：

`python_ai_client.go` 中将 `AIToolResponse` 改为与 Python 实际返回一致的结构：
```go
type AIToolResponse struct {
    TaskID    string `json:"task_id"`
    Status    string `json:"status"`
    ModelName string `json:"model_name"`
    ResultURL string `json:"result_url"`
    CreatedAt string `json:"created_at"`
    LatencyMs int    `json:"latency_ms"`
}
```

### #3 任务轮询 get_task_status 为 TODO 占位 🟢

**原严重度**：🔴 → **现严重度**：🟢（GenAI 代理同步/流式返回，不需要轮询）

**现象**：前端轮询 `GET /api/v1/ai-tools/tasks/:task_id` 永远返回 `status: "not_found"`，导致所有依赖轮询的功能无法展示结果。

**位置**：`ai-server/app/api/v1/ai_tools.py` L456-480

```python
@router.get("/tasks/{task_id}", response_model=TaskResponse)
async def get_task_status(task_id: str, ...):
    # TODO: 从数据库查询任务状态
    return {
        "id": 0,
        "task_type": "unknown",
        "model_name": "",
        "status": "not_found",
        "progress": 0,
        "result_url": None,
    }
```

**根本原因**：当前 AI 任务是同步执行的（Python 阻塞等待结果），创建任务时没有写入数据库，轮询自然查不到。

**影响范围**：仅 ai-tools 路径。GenAI 代理的 `generateContent` 和 `streamGenerateContent` 是同步/流式返回，不需要任务轮询机制。

**修复**（如仍需 ai-tools 路径）：

1. 任务创建时将任务信息写入数据库（任务表）。
2. `get_task_status` 从数据库查询并返回真实状态。
3. 若计划支持长耗时异步任务（如文生视频），需引入任务队列（Celery/RQ）。

## 🟡 中等问题

### #4 /auth/me 的 user_type 类型与登录/注册不一致（运行时 panic 风险）

**严重度**：🟡（保持不变，影响所有路径）

**现象**：`/auth/me` 运行时可能 panic，因为类型断言不匹配。

**位置**：

| 来源 | 文件:行 | 类型 |
|------|---------|------|
| JWT Claims | `pkg/jwt/jwt.go` L14 | `uint8` |
| 登录/注册响应 UserInfo | `service/auth_service.go` L53 | `uint8` |
| auth 中间件 c.Set | `middleware/auth.go` L52 | `claims.UserType`（uint8） |
| /auth/me handler | `handler/auth_handler.go` L109 | `userType.(string)` ← 类型断言为 string |

`auth.go` 中间件 `c.Set("user_type", claims.UserType)` 存的是 `uint8`，但 `auth_handler.go` L109 做 `userType.(string)` 断言，**uint8 无法断言为 string，运行时会 panic**。

**修复**：统一类型。

方案 A（推荐）— handler 也用 uint8：
```go
// auth_handler.go UserInfoResponse
type UserInfoResponse struct {
    ID       uint64 `json:"id"`
    Username string `json:"username"`
    Email    string `json:"email"`
    UserType uint8  `json:"user_type"`
    Credits  int32  `json:"credits"`
}
```

方案 B — 全部改用 string（需同时改 JWT Claims 和 service 层）。

### #9 Go 验证器比 Python 更严格（部分字段必填不一致）

**严重度**：🟡（仅影响 ai-tools 旧路径）

**现象**：前端按 Python 文档省略"可选"字段（如 `width`、`height`、`steps`），会被 Go 网关返回 400。

**位置**：`ai_tools_handler.go` L36-92（Go binding） vs `ai-server/app/schemas/task.py` L54-100（Python Pydantic）

| 接口 | 字段 | Go（binding tag） | Python |
|------|------|-------------------|--------|
| text-to-image | `width` | `binding:"required"` | 默认 1024，可选 |
| text-to-image | `height` | `binding:"required"` | 默认 1024，可选 |
| text-to-image | `steps` | `binding:"required"` | 默认 30，可选 |
| text-to-image | `cfg_scale` | `binding:"required"` | 默认 7，可选 |
| text-to-video | `width` | `binding:"required"` | 默认 1024，可选 |
| text-to-video | `height` | `binding:"required"` | 默认 1024，可选 |
| text-to-video | `steps` | `binding:"required"` | 默认 30，可选 |
| inpaint | `width`/`height`/`steps`/`cfg_scale` | `binding:"required"` | 有默认值，可选 |
| frame-to-video | `steps` | `binding:"required"` | 默认 30，可选 |

**修复**：将 Go 验证器中这些字段的 `binding:"required"` 改为 `binding:"omitempty,min=...,max=..."`，并在 Go handler 中设置默认值（或透传给 Python 让 Python 处理默认值）。
